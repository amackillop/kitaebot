# NixOS VM

## Purpose

The agent runs inside a NixOS virtual machine. This provides isolation, reproducibility, and a consistent deployment target whether running locally or in the cloud.

## Why a VM?

1. **Isolation** ‚Äî Agent can't affect host system
2. **Reproducibility** ‚Äî Same environment everywhere
3. **Security** ‚Äî VM is the security boundary, not regex guards
4. **Simplicity** ‚Äî One artifact to deploy, backup, restore

## Directory Structure

The VM module lives in `vm/`, deployed via `deploy/`:

```
vm/
‚îî‚îÄ‚îÄ configuration.nix      # NixOS module with kitaebot options

deploy/
‚îú‚îÄ‚îÄ flake.nix              # Deployment flake (imports VM module)
‚îú‚îÄ‚îÄ flake.lock
‚îî‚îÄ‚îÄ configuration.nix      # Local dev overrides
```

The root `flake.nix` exports the module as `nixosModules.vm` and the binary as `packages.default`.

## VM Module

The `configuration.nix` is a NixOS module that:

- Imports `qemu-vm.nix` for QEMU-based virtualisation
- Defines `kitaebot.package` option (required ‚Äî the kitaebot binary)
- Defines `kitaebot.sshKeys` option (list of SSH public keys for root)
- Defines `kitaebot.dev` option (mounts host nix store for faster dev builds)
- Defines `kitaebot.secretsDir` option (directory containing one file per credential)
- Defines `kitaebot.gitConfig` option (`{ name, email, signingKey? }`) ‚Äî identity and signing flow into `programs.git`
  - When `signingKey` is set: imports GPG key via `ExecStartPre`, configures `commit.gpgsign`, uses absolute Nix store path for `gpg.program` (not on exec tool PATH), sets `GNUPGHOME`
- Creates a dedicated `kitaebot` system user and group
- Provisions workspace directories via tmpfiles (`/var/lib/kitaebot/{sessions,locks,memory,projects}`)
- Runs `kitaebot run` as a long-lived systemd service
- Configures: OpenSSH (key-only, no password), firewall (port 22), base packages
- VM specs: 1024MB RAM, 2 cores, port forwarding (host 2222 ‚Üí guest 22)

## Daemon Service

The daemon runs as a long-lived systemd service:

```ini
# kitaebot.service (generated by NixOS module)
[Unit]
Description=Kitaebot agent daemon
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
ExecStart=/usr/bin/kitaebot run
User=kitaebot
WorkingDirectory=/var/lib/kitaebot
LoadCredential=provider-api-key:/var/lib/kitaebot-secrets/provider-api-key
LoadCredential=telegram-bot-token:/var/lib/kitaebot-secrets/telegram-bot-token
# Conditional:
# LoadCredential=github-token:...       (when github.enabled)
# LoadCredential=gpg-signing-key:...    (when gitConfig.signingKey set)
Environment=KITAEBOT_WORKSPACE=/var/lib/kitaebot
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
```

The daemon manages its own heartbeat timer internally ‚Äî no separate systemd timer needed.

## systemd Hardening

Both the heartbeat and daemon units apply the following restrictions:

- **Process isolation** ‚Äî `ProtectProc=invisible`, `ProcSubset=pid`
- **Filesystem** ‚Äî `ProtectSystem=strict`, `ProtectHome=true`, `ReadWritePaths=/var/lib/kitaebot`, `PrivateTmp=true`
- **Privilege** ‚Äî `NoNewPrivileges=true`, empty `CapabilityBoundingSet`
- **Syscalls** ‚Äî `@system-service` filter, `~@privileged`, `~@resources`, native arch only
- **Network** ‚Äî `AF_INET`, `AF_INET6`, `AF_UNIX` only
- **Kernel** ‚Äî All `Protect*` tunables enabled, `LockPersonality`, `RestrictNamespaces`, `MemoryDenyWriteExecute`

Verify with: `systemd-analyze security kitaebot-heartbeat`

## Building and Running

```bash
# Build VM image
nix build ./deploy

# Run VM
./result/bin/run-kitaebot-vm

# SSH in
ssh -p 2222 root@localhost
```

## Verification

After booting:

```bash
systemctl status kitaebot          # daemon running
id kitaebot                        # dedicated user exists
ls -la /var/lib/kitaebot/          # workspace dirs exist
ls /var/lib/kitaebot/sessions/     # session files (after first message)
journalctl -u kitaebot -f          # follow daemon logs
```

To test the REPL:

```bash
su - kitaebot
kitaebot chat
```

## Dev Mode

With `kitaebot.dev = true`, the VM mounts the host's nix store and uses a tmpfs writable store. This avoids re-downloading packages during development.

## Secrets Management

Secrets are loaded from credential files, not environment variables. See [spec 13](13-credentials.md) for the full design.

The NixOS module uses `LoadCredential=` to provision secrets. systemd copies files from `kitaebot.secretsDir` to `/run/credentials/<unit>/` with mode 0400, sets `CREDENTIALS_DIRECTORY` automatically. The process reads files at startup; secrets never enter the environment.

Required credential files:

| File | Purpose | Conditional |
|------|---------|-------------|
| `provider-api-key` | LLM provider authentication | Always |
| `telegram-bot-token` | Telegram Bot API token | Always |
| `github-token` | GitHub PAT for clone/push/PRs | `github.enabled` |
| `gpg-signing-key` | GPG private key for commit signing | `gitConfig.signingKey` set |

**Local dev** ‚Äî The deploy config mounts the repo's `secrets/` directory into the guest at `/mnt/kitaebot-secrets` via QEMU 9p. One file per secret:

```bash
echo 'sk-or-...' > secrets/provider-api-key
echo '0000000000:...' > secrets/telegram-bot-token
echo 'ghp_...' > secrets/github-token
gpg --export-secret-keys --armor KEY_ID > secrets/gpg-signing-key
```

The `secrets/` directory contents are gitignored (only `.gitkeep` is tracked). The mount target must be under `/mnt`, not `/run` (tmpfs shadows initrd mounts on `/run`).

**Production** ‚Äî Provisioned via sops-nix or manual deployment:

```nix
kitaebot.secretsDir = "/var/lib/kitaebot-secrets";
```

No secrets in the repo. No secrets in the nix store. No secrets in the process environment.

## Future Considerations

- **bubblewrap** ‚Äî Per-command namespace isolation for exec tool
- **cgroup limits** ‚Äî `TasksMax`, `MemoryMax` on systemd unit
- **seccomp-bpf** ‚Äî In-process syscall filtering (covers `kchat` path)
- **Resource limits** ‚Äî `MemoryMax`, `CPUQuota` via systemd
- **Cloud deployment** ‚Äî nixos-anywhere, AMI/GCE images
- **Container option** ‚Äî For users who prefer Docker
- **sops-nix** ‚Äî Encrypted secrets at rest in the repo
- **Watchdog** ‚Äî systemd `WatchdogSec` for daemon health monitoring
