# NixOS VM

## Purpose

The agent runs inside a NixOS virtual machine. This provides isolation, reproducibility, and a consistent deployment target whether running locally or in the cloud.

## Why a VM?

1. **Isolation** — Agent can't affect host system
2. **Reproducibility** — Same environment everywhere
3. **Security** — VM is the security boundary, not regex guards
4. **Simplicity** — One artifact to deploy, backup, restore

## Directory Structure

The VM module lives in `vm/`, deployed via `deploy/`:

```
vm/
└── configuration.nix      # NixOS module with kitaebot options

deploy/
├── flake.nix              # Deployment flake (imports VM module)
├── flake.lock
└── configuration.nix      # Local dev overrides
```

The root `flake.nix` exports the module as `nixosModules.vm` and the binary as `packages.default`.

## VM Module

The `configuration.nix` is a NixOS module that:

- Imports `qemu-vm.nix` for QEMU-based virtualisation
- Defines `kitaebot.package` option (required — the kitaebot binary)
- Defines `kitaebot.sshKeys` option (list of SSH public keys for root)
- Defines `kitaebot.dev` option (mounts host nix store for faster dev builds)
- Defines `kitaebot.secretsFile` option (path to env file with API keys, optional)
- Creates a dedicated `kitaebot` system user and group
- Provisions workspace directories via tmpfiles (`/var/lib/kitaebot/{memory,projects}`)
- Runs `kitaebot heartbeat` via systemd timer (5min after boot, every 30min)
- Configures: OpenSSH (key-only, no password), firewall (port 22), base packages
- VM specs: 1024MB RAM, 2 cores, port forwarding (host 2222 → guest 22)

## Heartbeat Service

The heartbeat runs as a systemd oneshot service triggered by a timer:

- **Service**: `kitaebot-heartbeat.service` — runs `kitaebot heartbeat` as the `kitaebot` user
- **Timer**: `kitaebot-heartbeat.timer` — 5min after boot, then every 30min, persistent
- **Environment**: `KITAEBOT_WORKSPACE=/var/lib/kitaebot`, API keys via `EnvironmentFile`
- **Working directory**: `/var/lib/kitaebot`

## Building and Running

```bash
# Build VM image
nix build ./deploy

# Run VM
./result/bin/run-kitaebot-vm

# SSH in
ssh -p 2222 root@localhost
```

## Verification

After booting:

```bash
systemctl list-timers                    # kitaebot-heartbeat.timer present
id kitaebot                              # dedicated user exists
ls -la /var/lib/kitaebot/                # workspace dirs exist
systemctl start kitaebot-heartbeat       # manual trigger
journalctl -u kitaebot-heartbeat         # check logs
```

## Dev Mode

With `kitaebot.dev = true`, the VM mounts the host's nix store and uses a tmpfs writable store. This avoids re-downloading packages during development.

## Secrets Management

Secrets are provided via `kitaebot.secretsFile` — a path to an env file inside the guest. The module doesn't care how the file gets there.

**Local dev** — The deploy config mounts the repo's `secrets/` directory into the guest at `/mnt/kitaebot-secrets` via QEMU 9p. Update the `sharedDirectories.source` path in `deploy/configuration.nix` to match your checkout.

```bash
echo 'OPENROUTER_API_KEY=sk-or-...' > secrets/.env
```

The `secrets/` directory contents are gitignored (only `.gitkeep` is tracked). The mount target must be under `/mnt`, not `/run` (tmpfs shadows initrd mounts on `/run`).

**Production** — Provisioned via cloud-init, sops-nix, or manual deployment:

```nix
kitaebot.environmentFile = "/var/lib/kitaebot/.env";
```

No secrets in the repo. No secrets in the nix store.
