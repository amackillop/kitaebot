# Spec 09: NixOS VM

## Motivation

The agent runs inside a NixOS virtual machine. This provides isolation,
reproducibility, and a consistent deployment target. The VM is the outermost
security boundary.

## Behavior

### Module Options

The NixOS module (`vm/configuration.nix`) exposes these options:

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `package` | package | (required) | The kitaebot binary |
| `sshKeys` | list of str | `[]` | SSH public keys for root access |
| `dev` | bool | `false` | Enable dev mode (adds debug packages) |
| `promptsDir` | path | `./prompts` | Directory containing SOUL.md, AGENTS.md, USER.md, HEARTBEAT.md |
| `secretsDir` | path | `/var/lib/kitaebot-secrets` | Directory containing one file per credential |
| `settings` | TOML attrset | `{}` | Serialized to `config.toml` in the workspace |
| `logLevel` | str | `"kitaebot=info"` | `RUST_LOG` filter string |
| `tools` | list of package | `[]` | Packages whose `bin/` dirs are on the service's PATH |
| `gitConfig` | null or `{ name, email, signingKey? }` | `null` | Git identity and optional GPG signing |
| `egressAllowlist` | list of str | (see egress filter) | Domains the kitaebot process may connect to |
| `dnsUpstream` | str | `"9.9.9.9"` | Upstream DNS resolver (Quad9) |
| `vm.memorySize` | int | 4096 | VM memory in MB |
| `vm.cores` | int | 4 | Virtual CPU cores |
| `vm.diskSize` | int | 20480 | Root disk size in MB |

### Workspace Provisioning

tmpfiles rules create directories and symlink files into the workspace:

**Directories** (owned by `kitaebot:kitaebot`, mode 0750):
- `/var/lib/kitaebot` (workspace root)
- `/var/lib/kitaebot/memory`
- `/var/lib/kitaebot/projects`
- `/var/lib/kitaebot/.config/direnv`

Note: `sessions/` is created by the Rust binary at startup, not by tmpfiles.

**Symlinks** (into the workspace from the Nix store):
- `config.toml` — serialized from `settings`
- `SOUL.md`, `AGENTS.md`, `USER.md`, `HEARTBEAT.md` — from `promptsDir`
- `.config/direnv/direnv.toml` — suppresses noisy direnv output

### Daemon Service

The daemon runs as a systemd service (`Type=simple`):

- **ExecStart**: `kitaebot run`
- **User/Group**: `kitaebot` (static UID 900 for nftables rules)
- **WorkingDirectory**: `/var/lib/kitaebot`
- **RuntimeDirectory**: `kitaebot` (creates `/run/kitaebot/` for the socket)
- **Restart**: `on-failure`, 10s delay
- **TasksMax/LimitNPROC**: 4096
- **Environment**: `KITAEBOT_WORKSPACE`, `RUST_LOG`, `PATH` (restricted to
  `tools` packages' bin dirs)

**Credentials** via `LoadCredential`:

| Credential | Conditional |
|------------|-------------|
| `provider-api-key` | Always |
| `telegram-bot-token` | Always |
| `github-token` | `git.enabled` or `github.enabled` |
| `gpg-signing-key` | `gitConfig.signingKey` set |

**Service ordering**: `nftables → dnsmasq → kitaebot`. DNS filtering must be
active before kitaebot makes any outbound connections.

### systemd Hardening

| Category | Directives |
|----------|-----------|
| Process isolation | `ProtectProc=invisible`, `ProcSubset=pid` |
| Filesystem | `ProtectSystem=strict`, `ProtectHome=true`, `ReadWritePaths=/var/lib/kitaebot`, `PrivateTmp=true` |
| Privilege | `NoNewPrivileges=true`, empty `CapabilityBoundingSet` and `AmbientCapabilities` |
| Syscalls | `@system-service`, `~@privileged`, native arch only |
| Network | `AF_INET`, `AF_INET6`, `AF_UNIX` only |
| Kernel | All `Protect*` tunables enabled, `LockPersonality`, `RestrictNamespaces`, `RestrictRealtime`, `RestrictSUIDSGID`, `MemoryDenyWriteExecute` |

### GPG Signing

When `gitConfig.signingKey` is set:

1. `ExecStartPre` imports the GPG private key from credentials into
   `GNUPGHOME=/var/lib/kitaebot/.gnupg`
2. `programs.git` sets `commit.gpgsign = true` and `gpg.program` to the
   absolute Nix store path (not on the exec tool's PATH — prevents the agent
   from calling gpg directly)

### Egress Filtering

DNS-based egress control via dnsmasq + nftables. See
[spec 18](18-egress-filter.md) for the full design.

- dnsmasq listens on `127.0.0.2`, resolves only allowlisted domains via
  upstream, returns NXDOMAIN for everything else
- nftables DNATs kitaebot UID's DNS to `127.0.0.2`, allows TCP 443 only to
  IPs resolved by dnsmasq, drops and logs everything else

### Direnv

`programs.direnv` is enabled with `nix-direnv`. The Rust process's
`DirenvCache` calls `direnv export json` and caches the result (see
[spec 03](03-tools.md#direnv-cache)).

### Dev Mode

When `dev = true`, debug packages are added to the system: `kchat` (connects
to `/run/kitaebot/chat.sock`), `vim`, `curl`, `dig`, `htop`.

### VM Configuration

- QEMU-based virtualisation, headless (`graphics = false`)
- Port forwarding: host 2222 → guest 22
- OpenSSH: key-only, no password authentication
- Firewall: port 22 open

## Boundaries

### Owns

- VM resource allocation (memory, cores, disk)
- System user/group creation (static UID 900)
- Workspace directory provisioning
- Prompt file and config symlinks
- systemd service definition and hardening
- Egress filtering infrastructure (dnsmasq + nftables)
- GPG signing flow
- Dev mode packages
- SSH configuration

### Does Not Own

- Agent behavior — the Rust binary handles that
- Credential content — provisioned externally (sops-nix or manual)
- Prompt file content — defined in `promptsDir`
- Landlock sandbox — applied inside the Rust binary at startup

## Failure Modes

| Failure | Behavior |
|---------|----------|
| Missing required secret | Service fails to start (`LoadCredential` error) |
| Missing prompt file | Symlink is dangling; workspace `system_prompt()` warns and skips |
| nftables/dnsmasq failure | Service ordering prevents kitaebot from starting |
| Sandbox unsupported | Rust binary warns, continues without Landlock |

## Constraints

- Static UID 900 is required for nftables rules to match the kitaebot process
- Secrets must exist at `secretsDir` before the service starts
- `tools` packages are only on the service's PATH, not the system shell
- The daemon manages its own heartbeat timer internally — no separate systemd
  timer
