# NixOS VM

## Purpose

The agent runs inside a NixOS virtual machine. This provides isolation, reproducibility, and a consistent deployment target whether running locally or in the cloud.

## Why a VM?

1. **Isolation** — Agent can't affect host system
2. **Reproducibility** — Same environment everywhere
3. **Security** — VM is the security boundary, not regex guards
4. **Simplicity** — One artifact to deploy, backup, restore

## Directory Structure

The VM configuration lives in `vm/`, separate from the main flake:

```
vm/
├── flake.nix              # VM-specific flake (nixpkgs only)
├── flake.lock
└── configuration.nix      # NixOS module with kitaebot options
```

The root `flake.nix` exports the module as `nixosModules.vm` for reuse.

## VM Module

The `configuration.nix` is a NixOS module that:

- Imports `qemu-vm.nix` for QEMU-based virtualisation
- Defines `kitaebot.sshKeys` option (list of SSH public keys for root)
- Defines `kitaebot.dev` option (mounts host nix store for faster dev builds)
- Configures: OpenSSH (key-only, no password), firewall (port 22), base packages (vim, git, curl, htop)
- VM specs: 1024MB RAM, 2 cores, port forwarding (host 2222 → guest 22)

Root login only, no separate `kitaebot` user yet. Agent service and workspace provisioning are not yet defined in the VM config.

## Building the VM

```bash
cd vm
nix build .#nixosConfigurations.kitaebot.config.system.build.vm
./result/bin/run-kitaebot-vm
```

## Running Locally

```bash
# Start VM
cd vm && nix run

# SSH in
ssh -p 2222 root@localhost
```

## Dev Mode

With `kitaebot.dev = true`, the VM mounts the host's nix store and uses a tmpfs writable store. This avoids re-downloading packages during development.

## Root Flake Integration

The root `flake.nix` handles the Rust build (via crane + fenix) and exports:
- `checks` — clippy, fmt, test
- `devShells.default` — dev shell with just, rust-analyzer, nix tooling

The VM module is exported as `nixosModules.vm` but the root flake doesn't currently define a `nixosConfiguration` — that lives in `vm/flake.nix`.

## Secrets Management

API keys should not be in the Nix configuration. Options:

**Option 1: Environment file**
```bash
# /var/lib/kitaebot/secrets.env
OPENROUTER_API_KEY=sk-or-...
```

**Option 2: sops-nix** (more secure)
```nix
sops.secrets.openrouter-api-key = {
  owner = "kitaebot";
};
```

## Future Considerations

- **Agent service** — systemd service running kitaebot as a daemon
- **Workspace provisioning** — systemd tmpfiles + activation script for template files
- **Dedicated user** — `kitaebot` user with restricted permissions, systemd hardening (ProtectSystem, ProtectHome)
- **Heartbeat timer** — systemd timer triggering periodic agent runs
- **Resource limits** — MemoryMax, CPUQuota via systemd
- **Cloud deployment** — nixos-anywhere, AMI/GCE images
- **Container option** — For users who prefer Docker
